## `parch` — Markdown specification

### Status

* **Type:** CLI utility
* **Scope:** Local machine, manual invocation (no daemon)
* **Primary integration:** Pueue CLI (required)

---

## 1. Overview

`parch` persists Pueue task metadata and logs to an external archive so you can safely remove tasks from Pueue (e.g., `pueue clean`) without losing historical output.

Key properties:

* **Source of truth:** `pueue log --json` (includes metadata + combined output)
* **No reliance on Pueue numeric IDs** for identity (IDs may reset)
* **One-file-per-task archive**, stored under an archive directory
* **Always-up-to-date UX:** every user-facing command runs a **fast implicit sync** by default

---

## 2. Terminology

* **Task (Pueue task):** A job known to Pueue.
* **Archived task:** A Pueue task snapshot persisted by `parch`.
* **Archive ID:** Stable ID generated by `parch` (used in this tool’s UI).
* **Fingerprint:** Deterministic hash used to deduplicate tasks across Pueue ID reuse.
* **Implicit sync:** A fast sync performed automatically before executing any command (default).

---

## 3. Requirements

### 3.1 Functional requirements

1. **Sync archive from Pueue**

   * Read Pueue tasks via `pueue log --json`.
   * Persist newly observed tasks into the archive.
   * Update previously archived tasks if metadata/output changed.

2. **Fast implicit sync**

   * By default, run `sync` before **every** command (`list`, `show`, `clean`, etc.).
   * Sync must be optimized to avoid rewriting unchanged task files.

3. **Browsing and filtering**

   * List archived tasks as a table.
   * Filter by:

     * date range (from/to, since)
     * relative windows (today, last-week)
     * status (success/failed/stopped/running/queued/…)
     * group
     * cwd/path substring
     * command substring (or regex if implemented)

4. **Viewing outputs**

   * Show stored combined output for a specific archived task.
   * Preserve ANSI escape codes by default (colors/progress output).

5. **Optional clean wrapper**

   * Provide a subcommand that does:

     * implicit sync (or an explicit “sync now”)
     * then runs `pueue clean` with forwarded arguments

### 3.2 Non-goals

* Not a scheduler, not a Pueue replacement.
* No guaranteed separation of stdout/stderr unless Pueue JSON provides it separately.
* No background service.

## 4. Archive storage

### 4.1 Default locations

* **Archive directory (default):**

  * `~/.local/share/parch/`
* **Config directory (optional):**

  * `~/.config/parch/`

(If desired later, this can be made fully XDG-compliant, but the above default is sufficient and stable.)

### 4.2 Layout

```
~/.local/share/parch/
  tasks/
    <archive_id>.json
    <archive_id>.json
  index.jsonl
  fingerprints.json
  lock
```

* `tasks/`: one JSON file per archived task
* `index.jsonl`: append-friendly index for fast listing (can be rebuilt)
* `fingerprints.json`: mapping `{fingerprint -> archive_id}` for O(1) dedup
* `lock`: advisory lock for concurrency control

### 4.3 Archive ID

* **Recommended:** UUIDv7 (time-sortable)
* Used for filenames and CLI references:

  * `tasks/<archive_id>.json`

### 4.4 Fingerprint (dedup key)

A task fingerprint is a deterministic hash over a stable subset of the Pueue JSON fields.

**Definition (normative):**

```
fingerprint = sha256(
  command + "\0" +
  cwd + "\0" +
  group + "\0" +
  created_at + "\0"
)
```

* `created_at` should be the highest-resolution creation timestamp available from Pueue JSON.
* If `created_at` is missing, fall back to the best available stable timestamp field(s) and include them in the hash input.
* `pueue_task_id` **MUST NOT** be part of the fingerprint.

### 4.5 Stored task file schema (`tasks/<archive_id>.json`)

Minimum required keys:

```json
{
  "archive_id": "01J... (uuidv7)",
  "fingerprint": "hex_sha256",
  "archived_at": "2026-02-28T19:22:11Z",

  "source": {
    "pueue_task_id": "12",
    "pueue_group": "default",
    "pueue_version": "optional"
  },

  "meta": {
    "command": "...",
    "cwd": "...",
    "status": "...",
    "timestamps": {
      "created_at": "...",
      "start_at": "...",
      "end_at": "..."
    }
  },

  "output": {
    "combined": "raw string with ANSI",
    "combined_sha256": "hex_sha256"
  },

  "raw": {
    "pueue_log_json": { }
  }
}
```

Notes:

* Storing `raw.pueue_log_json` is recommended for forward compatibility.
* `combined_sha256` enables “did output change?” checks without expensive comparisons.

### 4.6 Index schema (`index.jsonl`)

Each line is one JSON object with fields used for sorting/filtering, e.g.:

```json
{
  "archive_id": "01J...",
  "fingerprint": "hex_sha256",
  "status": "success",
  "group": "default",
  "cwd": "/home/dev/project",
  "command": "uv run ...",
  "created_at": "...",
  "start_at": "...",
  "end_at": "...",
  "pueue_task_id": "12"
}
```

Index rules:

* Index is **rebuildable** by scanning `tasks/`.
* Implementations may choose either:

  * append-only “latest-wins” lines, or
  * rewrite-on-update with atomic replacement.
* The spec does not require one approach; correctness matters more than format.

## 5. Sync semantics

### 5.1 Implicit sync (default behavior)

**Normative behavior:**

* Every command performs an implicit sync **before** its main action, unless disabled by a global flag.

**Global flag:**

* `--no-sync`: skip implicit sync for this invocation

**Rationale:**

* Ensures “logs are always up to date” without requiring manual `sync`.

### 5.2 Explicit sync command

* `parch sync` performs a full sync and exits.

### 5.3 Fast sync requirements

Sync should be “fast” in the common case (few/no changes). Minimum required optimizations:

1. **Dedup lookup is O(1)**

   * Load `fingerprints.json` once per invocation.
   * Use it to decide “new task vs existing task” without scanning `tasks/`.

2. **No rewrite if unchanged**

   * For known fingerprints, compute a cheap “change detector”:

     * status change, or
     * metadata change, or
     * `combined_sha256` change
   * Only rewrite the task file if something actually changed.

3. **Atomic writes**

   * Write temp file then rename.

4. **Concurrency safety**

   * Acquire an advisory lock for sync (e.g., `flock` or lockfile protocol).
   * If lock cannot be acquired quickly:

     * either wait briefly (configurable) or fail with a clear error.

### 5.4 Sync scope

Optional tuning (implementation-defined but recommended):

* `--include-running` (default: `false`)

  * When false, only archive tasks in terminal states (success/failed/stopped/etc.).
  * When true, include running tasks and update them over time.

This can significantly reduce churn during frequent implicit sync.

## 6. CLI interface

### 6.1 Global options (apply to all commands)

* `--archive-dir <path>`: override archive root directory
* `--pueue-bin <path>`: path to `pueue` binary (default: `pueue`)
* `--sync / --no-sync`: enable/disable implicit sync (default: `--sync`)
* `--color auto|always|never` (default: `auto`)
* `--pager auto|always|never` (default: `auto`)
* `--quiet` / `--verbose`

### 6.2 Commands

#### `parch sync`

Fetch current `pueue log --json`, merge into archive.

Options:

* `--include-running`
* `--timeout <secs>` (optional; for invoking Pueue)

Exit codes:

* `0` success
* `2` failed to invoke pueue / pueue returned error
* `3` invalid/unexpected JSON
* `4` archive IO error (permissions, disk full, etc.)

---

#### `parch list`

List archived tasks, default format: table.

Options (filters):

* Time:

  * `--from <date|datetime>`
  * `--to <date|datetime>`
  * `--since <duration>` (e.g., `7d`, `24h`)
  * `--today`, `--yesterday`, `--last-week`
* Status:

  * `--status <value>` (repeatable)
* Group:

  * `--group <name>` (repeatable)
* Text:

  * `--cmd <substring>` (or regex if implemented)
  * `--cwd <substring>`
* Output format:

  * `--format table|json` (default: `table`)
* Sorting:

  * `--sort created|start|end` (default: `end` if present else `created`)
  * `--reverse`
  * `--limit <n>`

**Timestamp used for time filtering:**

* Prefer `end_at`, else `start_at`, else `created_at`.

---

#### `parch show <archive_id>`

Print the stored output for a specific archived task.

Options:

* `--meta` (prepend metadata header)
* `--raw-json` (dump full stored task JSON)
* `--no-ansi` (strip ANSI escapes from output)
* `--no-pager` (force no pager)

Default:

* Output shown via pager when interactive and long.
* ANSI preserved.

---

#### `parch clean -- [pueue-clean-args…]`

Convenience wrapper.

Behavior:

1. Perform implicit sync (unless `--no-sync`)
2. Execute `pueue clean` with forwarded args (everything after `--`)

Example:

```bash
parch clean -- --successful --group default
```

## 7. Configuration file (optional)

`~/.config/parch/config.toml` (example):

```toml
archive_dir = "~/.local/share/parch"
pueue_bin = "pueue"
include_running = false
color = "auto"
pager = "auto"
```

Priority:

1. CLI flags
2. Config file
3. Defaults

---

## 8. Output preservation requirements

* Store the `pueue log --json` output field(s) **verbatim**.
* ANSI escape sequences **MUST** be preserved by default.
* Any “rendering” (color stripping, paging) happens only at display time.

---

## 9. Implementation notes (non-normative but recommended)

* Use a streaming JSON parser if `pueue log --json` is large.
* Keep `fingerprints.json` compact and rewrite atomically on updates.
* Provide `parch rebuild-index` (optional) to regenerate `index.jsonl` from `tasks/`.

---

## 10. Examples

Archive implicitly and list:

```bash
parch list --today
```

Show a task:

```bash
parch show 01JZ... --meta
```

Sync only:

```bash
parch sync
```

Archive-then-clean:

```bash
parch clean -- --all
```

Skip implicit sync (e.g., offline browsing):

```bash
parch --no-sync list --status failed
```
